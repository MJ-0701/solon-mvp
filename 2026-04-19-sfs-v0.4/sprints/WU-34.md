# WU-34 — Codex CLI `/sfs` Compatibility Layer

## §0. Problem

Solon public API is `/sfs`.

Codex desktop app / compatible Codex surfaces can deliver `/sfs ...` as prompt
text to the model/Skill. That path is valid and was reinforced in
`0.5.5-product`.

codex-cli blocking builds do something different:

1. User types `/sfs status`.
2. CLI native slash parser handles the input before the model sees it.
3. `/sfs` is not a registered native command.
4. CLI returns `Unrecognized command '/sfs'`.
5. `.agents/skills/sfs/SKILL.md` never receives the message.

This is not a user invocation mistake. It is a runtime adapter compatibility
gap.

## §1. Current Mitigation

- Codex desktop app: `/sfs ...` remains canonical and first-class.
- Codex CLI blocking build: `$sfs ...`, `sfs ...`, natural language, or direct
  bash are temporary bypasses only.
- Optional user prompt fallback (`~/.codex/prompts/sfs.md`) remains legacy and
  must not be described as primary API.

## §2. Non-Goals

- Do not remove or downgrade Codex desktop app `/sfs`.
- Do not spend a full investigation pass on official Codex registration unless
  a new CLI release, docs update, or concrete user signal appears.
- Do not mark parity complete while codex-cli bare `/sfs` is blocked before the
  model/Skill.

## §3. Acceptance Criteria

Done means one of:

1. codex-cli in a real installed consumer project accepts `/sfs status` and
   reaches Solon bash SSoT; or
2. a concrete upstream/product blocker is recorded with tested CLI version,
   exact failure mode, and the best supported fallback path, while desktop app
   `/sfs` remains canonical.

## §4. Suggested Next Micro-Step

After Windows wrapper verification, capture the exact codex-cli version +
failure transcript in this WU and decide whether to wait for upstream support
or open a small compatibility issue/adapter note.
