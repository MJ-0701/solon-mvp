# WU-34 — Codex CLI SFS Invocation Policy

## §0. Problem

Solon has multiple runtime entry shapes. They are not all native slash commands.

Codex desktop app / compatible Codex surfaces can deliver `/sfs ...` as prompt
text to the model/Skill. That path is valid and was reinforced in
`0.5.5-product`.

Codex CLI does something different:

1. User types `/sfs status`.
2. CLI native slash parser handles the input before the model sees it.
3. `/sfs` is not a registered native command.
4. CLI returns `Unrecognized command '/sfs'`.
5. `.agents/skills/sfs/SKILL.md` never receives the message.

This is not a user invocation mistake or an unresolved Solon parity target.
Bare `/sfs` is not the supported Codex CLI shape. The official Codex CLI Skill
invocation is `$sfs ...`.

Windows PowerShell direct shell usage is separate from Codex CLI prompt usage:
use `sfs.cmd ...` for the global shell wrapper.

## §1. Current Mitigation

- Codex desktop app: `/sfs ...` remains canonical and first-class.
- Codex CLI: `$sfs ...` is the official Skill invocation.
- Windows PowerShell shell: `sfs.cmd ...` is the direct global wrapper command.
- POSIX shell / Git Bash / WSL: `sfs ...` is the direct global wrapper command.
- Optional user prompt fallback (`~/.codex/prompts/sfs.md`) remains legacy and
  must not be described as primary API.

## §2. Non-Goals

- Do not remove or downgrade Codex desktop app `/sfs`.
- Do not promise bare `/sfs` support in Codex CLI.
- Do not describe official `$sfs ...` as a temporary bypass.
- Do not spend a full investigation pass on native slash registration unless a
  future Codex CLI release or concrete user signal changes the official shape.

## §3. Acceptance Criteria

Done means:

1. Product docs describe Codex CLI as `$sfs ...`, not bare `/sfs`.
2. Product docs describe Windows PowerShell direct shell usage as `sfs.cmd ...`.
3. The Codex Skill keeps `$sfs ...` / `sfs ...` first and treats `/sfs ...` only
   as text that may work if a host surface passes it through to the model.
4. No upstream issue is opened for bare `/sfs` unless official Codex CLI support
   changes or `$sfs ...` fails.
