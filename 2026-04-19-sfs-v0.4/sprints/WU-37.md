# WU-37 — Implement Execution Contract Hardening

## §0. Problem

`/sfs implement` currently has good AI coding guardrails, but the runtime
surface still leans too hard toward "code changes now". That makes SFS feel
developer-only even though the system is meant to serve founders, PMs,
operators, designers, taxonomy-heavy domains, QA, and infra work.

The product meaning should be:

> Implement = execute the smallest verified work slice and record evidence.

Code is one important execution artifact, not the only one.

## §1. Acceptance Criteria

1. `sfs-implement.sh` no longer says the AI runtime must "implement code now";
   it must execute the work slice and record evidence.
2. `implement.md` template names non-code artifacts as first-class outputs and
   keeps code-specific DDD/TDD guardrails conditional on code being touched.
3. Installed Codex Skill / legacy Codex prompt explain implement as execution
   contract, not code-only.
4. Implementation persona supports division outputs, not only changed code
   files/modules.
5. README / GUIDE / 10X-VALUE product language reflects non-developer usage.
6. Shell syntax checks and a small sandbox smoke for `/sfs implement` pass.

## §2. Notes

- Keep command name `/sfs implement` for compatibility.
- Do not introduce a second command unless future UX testing shows
  `/sfs execute` is needed as an alias.
- Preserve the existing backend transaction and security guardrails for code
  work; this WU broadens the outer contract.
