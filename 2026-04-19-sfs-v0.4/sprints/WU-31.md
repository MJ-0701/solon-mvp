# WU-31 — Release tooling Phase 0 (local sh, cut-release + sync-stable-to-dev)

> **Context**: 23rd 세션 사용자 결정 = 배포 자동화 옵션 β (로컬 sh 우선, GitHub Action 은 후속 Phase 1+2 별도 WU). 본 WU = spec only ("계획만 만들어 둬" 사용자 지시 정합). 실 bash 구현은 다음 세션 또는 사용자 컨펌 후.

## §0. 작업 진행 원칙

1. **MINIMAL change** — 옵션 β 채택. 사양 = 로컬 sh 3 개 + `.visibility-rules.yaml` allowlist. 추가 기능 신설 금지 (GitHub Action 은 별도 후속 WU).
2. **R-D1 (dev-first)** — 본 sh 들의 source-of-truth = `2026-04-19-sfs-v0.4/scripts/`. stable repo 는 sh 들을 동봉하지 않음 (사용자 운영 도구라서). 단 OSS 공개 시 fork user 도 활용 가능하도록 향후 결정 (decision_points (1)).
3. **§1.5 push 정합** — 모든 sh 는 commit 까지만, push 안 함. cut-release.sh 는 stable repo 에 commit + tag 만 만들고, 실 push 는 사용자 터미널 (CHANGELOG entry 에 명시).
4. **§1.6 FUSE bypass 자동 적용** — sh 안에서 git 작업 시 `.git/index.lock` 충돌 시 cp/rsync 우회 패턴 내장 또는 외부 처리.
5. **dry-run default** — `cut-release.sh` / `sync-stable-to-dev.sh` 모두 `--dry-run` default. `--apply` 명시 시만 실 변경.

## §2. `cut-release.sh` 사양 (정방향, dev → stable)

### 2.1 호출

```sh
bash 2026-04-19-sfs-v0.4/scripts/cut-release.sh \
  --version 0.3.0-mvp           # 필수: 새 release 의 semver
  --dry-run                     # default. 실 적용은 --apply
  [--allow-dirty]               # 선택: stable repo 에 uncommitted 변경 있어도 강행 (기본 차단)
  [--no-tag]                    # 선택: stable 에 git tag 안 만듦 (default = tag 생성)
```

### 2.2 동작 (단계별)

1. **Pre-flight**:
   - dev staging path = `2026-04-19-sfs-v0.4/solon-mvp-dist/` 존재 확인
   - stable path = `~/workspace/solon-mvp/` 존재 확인 + `.git` 정상
   - dev / stable 둘 다 `git status` clean (또는 `--allow-dirty` 명시)
   - `.visibility-rules.yaml` 읽기 — allowlist 적용 준비
2. **Visibility filter**:
   - dev staging 안의 파일 중 `oss-public` tier 만 선별 (`.visibility-rules.yaml` 의 `include_only_if.tier == oss-public`)
   - 만약 frontmatter 없는 파일이면 **FAIL** (사용자 강제 결정 필요, leak 방지)
3. **Diff preview (dry-run 단계)**:
   - 선별된 파일들과 stable repo 의 동명 파일 diff 출력
   - 변경 파일 N개 / 라인 +A/-D / 신규 N개 / 삭제 N개 summary
   - 사용자가 dry-run 결과 확인
4. **Apply (`--apply` 시만)**:
   - `rsync` 로 dev staging → stable repo 복사 (visibility filter 적용)
   - stable repo 에서 VERSION 갱신 (인자 `--version` 적용)
   - CHANGELOG.md 에 entry 추가 (template 형식: `## [X.Y.Z-mvp] - YYYY-MM-DD`)
   - `git add -A` + `git commit -m "release: X.Y.Z-mvp"` (stable repo 안에서)
   - (옵션) `git tag vX.Y.Z-mvp` (default)
   - **push 안 함** — 사용자 터미널에서 `git push origin main && git push origin --tags`
5. **Post-flight**:
   - dev staging 의 VERSION + CHANGELOG 도 동일하게 bump (R-D1 정합 — 다음 사이클은 이 VERSION 기준)
   - dev repo 에서 `git commit -m "WU-31/cut(stable): vX.Y.Z-mvp <stable-sha>"` (stable commit sha 추적)
   - 본 PROGRESS.md ① Just-Finished 섹션에 release 기록 자동 append (후속 헬퍼)

### 2.3 Allowlist (현 시점 기준)

dev staging `solon-mvp-dist/` → stable `~/workspace/solon-mvp/` 로 가는 파일 set (visibility = `oss-public` 로 일괄 가정, 실 frontmatter 검증은 sh 안에서):

| 파일 | dev staging | stable | 비고 |
|---|---|---|---|
| VERSION | ✅ | ✅ | release 시 bump |
| CHANGELOG.md | ✅ | ✅ | release 시 entry |
| README.md | ✅ | ✅ | |
| CLAUDE.md | ✅ | ✅ | consumer 용 (templates/CLAUDE.md.template 와 다름) |
| install.sh | ✅ | ✅ | |
| upgrade.sh | ✅ | ✅ | |
| uninstall.sh | ✅ | ✅ | |
| templates/ | ✅ | ✅ | 디렉토리 통째로 |
| **APPLY-INSTRUCTIONS.md** | ✅ | ❌ | dev staging 전용 (운영자 instruction) — allowlist 에서 **제외** |

**제외 검증**: `APPLY-INSTRUCTIONS.md` 가 stable 에 절대 안 가야 함 (sh 안에서 hard-coded blocklist).

### 2.4 Checksum manifest

`solon-mvp-dist/.checksums.txt` (dev) + `~/workspace/solon-mvp/.checksums.txt` (stable) — 양쪽 동일하면 sync 일치. drift 감지의 backbone.

```
# 형식 (sha256 + relative path, sorted)
<sha256>  VERSION
<sha256>  CHANGELOG.md
...
```

cut-release.sh 가 release 후 양쪽에서 manifest 재생성 + commit.

## §3. `sync-stable-to-dev.sh` 사양 (역방향, hotfix path)

### 3.1 호출

```sh
bash 2026-04-19-sfs-v0.4/scripts/sync-stable-to-dev.sh \
  --stable-sha <commit-sha>     # 필수: stable repo 의 hotfix commit sha
  --dry-run                     # default
  [--apply]                     # 실 cp + git add
```

### 3.2 동작

1. Stable repo 에서 `<commit-sha>` 의 변경 파일 목록 추출 (`git show --name-only <sha>`)
2. 각 파일에 대해 stable → dev staging 복사 (allowlist 정합 검증)
3. dev staging 에서 `git add` 만 (commit 은 사용자 명시 — `WU-31/sync(stable): <sha>` 형식)
4. CHANGELOG.md 에 hotfix entry 자동 추가 (templates 별도)

**§1.13 R-D1 정합**: hotfix 는 stable 에서 발견 + 수정 OK, 단 같은 세션 내 dev staging back-port 의무. 본 sh 가 그 의무를 1 회 호출로 자동화.
