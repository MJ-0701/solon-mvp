# WU-35 — Code Development Team Hardening

## §0. Intent

Upgrade the code development team so backend implementation naturally carries
production-minded engineering discipline without turning every MVP task into a
large architecture review.

This WU is not a request to create a separate `security-guard` skill. It is a
request to make `/sfs implement` smarter.

  - Must become a backend transaction guardrail.

## §2. Operating Contract

### Always-on backend guardrails

Apply automatically when implementing backend code:

- Transaction boundaries belong in service/application layers.
- External API calls should not sit inside long DB transactions.
- `REQUIRES_NEW` requires explicit intent and connection-pool awareness.
- If `REQUIRES_NEW` changes state and the caller needs that result, return a
  result object instead of re-reading through the same outer EntityManager.
- Batch code must account for chunk transaction scope and JPA first-level cache.
- Idempotency, ordering, outbox, retries, and state history are mandatory design
  considerations when MQ, Webhook, partner API, payment, settlement, or batch
  effects appear.
- Hikari pool sizing must account for nested connection demand.
- Tests must match the risk: mocks for branch logic, persistence/integration
  tests for transaction/flush/commit/cache behavior, Batch tests for chunk flows.

### Trigger-based guardrails

Apply only when the implementation touches the trigger:

- Spring Batch -> chunk transaction, reader/processor/writer interaction,
  retry/skip/listener, job idempotency.
- Partner/external API -> timeout, retry with jitter, circuit breaker, bulkhead,
  fallback, contract, correlation ID.
- Event/MQ/Webhook -> outbox, idempotent consumer, ACK timing, DLQ,
  sequence/order, replay.
- State machine/status changes -> allowed transition matrix, append-only history,
  reconciliation path.
- Async/scheduler/thread pool -> executor sizing, MDC/SecurityContext propagation,
  graceful shutdown.

### Scale-gated guardrails

Ask one high-level question at sprint/project scope, not on every tiny edit:

```text
Security / Infra / DevOps guard level?
- light: secrets, authz/authn, sensitive data, dependency/container basics,
  actuator/public surface, logging hygiene
- full: threat model, AWS/IAM/network/WAF/audit, CI/CD gates, rollback,
  observability, runbooks
- skip: local prototype/MVP; keep basic hygiene only and record deferred risks
```

Default suggestion:

- Public API, money/PII, partner integration, or production deployment -> light.
- Regulated/payment/financial data or production AWS changes -> full.
- Local-only throwaway prototype -> skip plus basic hygiene.

## §3. MVP Filter

The guardrail should distinguish "must fix now" from "real but too large for
this sprint".

Use these buckets:

- `required-now`: correctness, money movement, PII leak, auth bypass, data loss,
  unrecoverable external side effect, release blocker.
- `light-now`: small configuration or test additions that reduce meaningful risk.
- `deferred`: valid architecture/infra/security item that is excessive for current
  MVP scope.
- `risk-accepted`: user or sprint explicitly accepts the risk for the current
  release.
