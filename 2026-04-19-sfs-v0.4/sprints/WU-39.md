# WU-39 — Context-Routing Structure for Solon Adapters

## §0. Goal

Make Solon read like a deep-module system:

- Entry docs are short routers.
- Context modules carry `summary` / `load_when` frontmatter.
- Agents read only the matching module for implement/review/release/upgrade/tidy/loop.
- Install/upgrade keeps the context modules in consumer projects.

## §1. Acceptance Criteria

1. `SFS.md`, `CLAUDE.md`, `AGENTS.md`, `GEMINI.md`, Claude command, Codex Skill,
   Codex prompt, and Gemini command route to `.sfs-local/context/`.
2. New context modules exist under `templates/.sfs-local-template/context/`.
3. `install.sh` copies context modules for both thin and vendored layouts.
4. `upgrade.sh` previews and updates context modules with backup safety.
5. Syntax checks and sandbox install/upgrade smoke pass.

## §2. Implementation Notes

- Adapter docs now act as routers instead of carrying full operating doctrine.
- `.sfs-local/context/` is the deep-module boundary: kernel + index + command/policy modules.
- `commands/implement.md` explicitly treats code as one valid artifact among taxonomy,
  design handoff, QA evidence, infra/runbook, decisions, docs, workflow, research,
  and user-facing operating material.
- Implement guardrails now include shared intent checks, domain language consistency,
  smallest-slice feedback, deep modules, gray-box delegation, and review handoff.
