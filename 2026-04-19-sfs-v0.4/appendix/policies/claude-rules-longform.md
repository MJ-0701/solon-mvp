# CLAUDE Rules — Longform (Appendix)

- 목적: `CLAUDE.md` 진입 비용(토큰) 절감.
- 규칙: **요약 + 구속력은 `CLAUDE.md`** 가 SSoT. 본 문서는 상세 절차/근거 보관.

13. **R-D1 (dev-first, stable sync-back)** — 배포 artifact (Solon docset `solon-mvp-dist/` staging ↔ 외부 `solon-mvp` stable repo) 수정은 **dev staging 에서 먼저** 하고 stable 은 staging cut 의 결과물만 수용한다. **예외 (hotfix path)**: 실 사용 중 stable 에서 발견된 버그는 stable 에서 수정 허용 — 단 ①같은 세션 안에 staging 에 동일 문안을 동기화 (cp + git add, **단순 string replace 가 아니라 `git show <sha>:<file>` 로 narrative 의도 read 후 통합** — P-14 mental coupling 안티패턴 차단), ②staging commit message 에 `sync(stable): <commit-sha>` 표기, ③다음 release 때 staging VERSION 을 stable VERSION 과 **skip 없이** 맞춘다. **①sync-back 누락 = 다음 cut 회귀 risk** (P-13 사고 패턴 = codex hotfix narrative 가 dev rsync 로 overwrite). 차단 = `cut-release.sh` Pre-flight 의 stable HEAD vs dev narrative key divergence 감지 (exit 7, `--allow-divergence` bypass). 근거: `learning-logs/2026-05/P-02-dev-stable-divergence.md` (2026-04-24 14 파일 reverse reconcile) + `P-13-dev-stable-divergence-on-cut.md` (2026-04-30 0.5.0-mvp 회귀 사고 + cut-release.sh 강화) + `P-14-mental-coupling-on-rename-fix.md` (2026-04-30 AI 측 read 누락 안티패턴). 자동화: `scripts/sync-stable-to-dev.sh` · `scripts/cut-release.sh` (P-13 divergence 감지 강화 완료).
14. **CLAUDE.md SSoT 분량 제약** — 본 CLAUDE.md 의 합산 line 수는 항상 **≤ 200 lines** 유지. 초과 시 가장 부록 성격 강한 § 을 별도 파일로 분리하고 link 1줄로 대체. 분리 우선순위 = 부록 > 본문 § / 라인 수 > 의미 밀도. 근거: 22nd 세션 `adoring-trusting-feynman` 사용자 결정 (가독성 + 진입 비용 최소화 + SSoT 1원 정합). 첫 분리 사례: §14 → `solon-status-report.md` (22nd 세션 적용, 의미 변경 0). 다른 § 후속 분리 시 본 §1.14 에 사례 append + 분리 사유 1줄 기록.
15. **자율진행 Pre-execution Review Gate** — AI 가 "큰 작업" (≥10분 / files_touched ≥3 / decision_points 신설 / spec change / visibility 등급 변경 중 하나) 자율진행 시작 전 **PLANNER(CEO, `agents/planner.md`) + EVALUATOR(CPO, `agents/evaluator.md`) 두 페르소나 review 의무**. 둘 다 PASS 시 사용자 (CEO final approver) 승인 요청 → 승인 받고 진행. 미승인 시 사유 보고 + 작은 단위 cascade (depth cap=3). 작은 작업 (heartbeat / log entry append / row 마킹 등) 은 review gate skip 자동 진행 OK. Spec SSoT: `tmp/sfs-loop-design.md §6.6` (24th-52 brave-gracious-mayer continuation 5 사용자 결정 + self-application worked example).

17. **결정 브리핑 7 단계 의무 형식** — AI 가 사용자에게 결정 요청 시 표 (table) 형식 금지. `tmp/sfs-briefing-template.md §2` 7 단계 prose 형식 의무 사용 (질문 / 배경 / 현재 상태 / 옵션 prose / 추천+사유 / 미결정 시 영향 / 답변 형식). 큰 결정 (framework / architecture) 은 +3 추가 단계 (4.5 prior art / 5.5 추천 사유 fact 출처 / 6.5 cascade). 근거: 24th-52 brave-gracious-mayer continuation 5 사용자 비판 ("표만 보고는 결정 못 한다") + self-application = 본 §1.17 신설 자체가 §2 형식 적용 첫 case.
18. **Git 실행/안내 형식 (copy-paste fallback, path/branch-neutral)** — §1.5/§1.22 의 git lifecycle 은 AI 가 실행한다. 실행이 권한/충돌/인증/보호 branch 로 막힐 때만 사용자에게 shell command block 을 제공한다: ① `cd <repo-root>` ② 필요한 `git status/add/commit/merge/push` ③ 현재 branch/target branch 명시. prose 형식 ("권장 message: <text>") 만 단독 제공 금지.

Spec SSoT: `sprints/WU-28.md`.
20. **단계적 안내 원칙 (Sequential Disclosure)** — AI 가 사용자에게 절차 안내 시 **(1) 정상 흐름** = 진단 또는 작업 1 step 후 **다음 1 step 만** 안내. 다다음 step / 옵션 분기 / 후속 plan 송출 금지 (다음 step 끝난 후 분리 송출). **(2) 문제 발견 흐름** = 진단 중 문제 발견 시 **즉시 stop + 그 문제만 안내 + 사용자 결정 또는 fix**. 다른 단계 송출 금지, 문제 해소 후에야 다음 단계 진입. **(3) 옵션 분기** = 결정 갈림길에서만 송출. 단순 정상 흐름에선 옵션 던지기 금지 (사용자 결정 부담 누적 방지). 근거 = 26th-1 admiring-compassionate-euler 사용자 verbatim 비판 ("진단도 단계가 여러개인데, 하나에서 문제가 발견되면 그걸 수정을 먼저 해야되는데 이때는 단계적으로 하나씩 절차대로 진행해야되는데 지금 쓸데없이 옵션이나 다음단계가 같이 튀어나옴 필요한 부분만 → 정상흐름일땐 다음단계 안내까지만"). bad case 2건 = sub-task 6.8 진입 시 진단+옵션(A/B/C/D)+다음단계 batch dump + stable repo 14 file 발견 시 진단+옵션(A/B/C)+분기(α/β) batch dump. §1.17 (7-step briefing prose) 의 cousin = "절차 안내" 영역 적용. dogfooding 사례 = `learning-logs/2026-05/P-11-sequential-disclosure.md` (worked example 3건 누적).
21. **운영 환경 매핑 (Multi-runtime role split)** — Claude Code / Cowork / Codex / Gemini-CLI 의 환경별 역할 분리는 [`RUNTIME-ABSTRACTION.md`](./RUNTIME-ABSTRACTION.md) §6.5 본체 참조. 본 §1 은 진입 인지만 보장 (도구 강제 X), 강제력은 §1.13 R-D1 + `scripts/cut-release.sh` divergence 감지 자동화가 담당. 도구 추가/변동 시 §1 mutate 0 — RUNTIME-ABSTRACTION §6.x adapter 슬롯 + §6.5 매핑 row 만 갱신. `/sfs loop` 운영은 기본 **single-runner**: 한 시점에 Codex/Claude/Cowork 중 하나만 loop runner 로 두고, 작업 성격별 최적 런타임 선택은 AI 가 판단한다. 병렬 loop 는 명시적 coordinator/분리 scope/의존성 없음이 확인된 예외에서만 허용한다. 근거: 26th 세션 `eager-festive-shannon` (2026-04-30) 사용자 (i)+(iii-b) 하이브리드 결정 + AGENTS bootstrap shim (v0.4-r4 변경점) + 2026-04-20 사용자 직접 지시 verbatim ("sfs를 claude 뿐만 아니라 codex랑 gemini-cli에서도 사용하고 싶거든??", WU-8) + 2026-05-01 Codex/Claude queue 병렬 dogfooding 결과 (진행 체감 저하, single-runner 운영 보정).
22. **Git Flow branch-first 운영 (project-wide notice)** — 사용자가 새 작업을 발화하면 AI 는 file edit 전에 `git status --short` + 현재 branch 를 확인하고, 작업 도메인을 판별해 `feature/<domain>-<short-slug>` 또는 production bugfix 는 `hotfix/<domain>-<short-slug>` 를 자동 생성/전환한다. branch 안 dirty 는 그 작업의 정상 진행 상태로 취급하되, 새 작업 시작 시 다른 도메인/untracked/legacy dirty set 이 섞여 있으면 자동 전환 금지 → stop + 분리/commit/stash 결정 요청. 완료 조건 = 검증 통과 + feature/hotfix commit + branch push + main 흡수(merge/PR 중 repo 정책에 맞는 경로) + origin main push. 근거: 2026-05-01 사용자 프로젝트 전체공지 ("사용자는 작업만 발화, AI 가 도메인별 branch 생성부터 main 흡수와 origin push 까지 수행").
23. **SFS 문서 생명주기 하네스** — `brainstorm/plan/implement/log/review` 는 진행 중 workbench(노트패드/화이트보드) 로만 두고, 완료 시 `/sfs report` 로 결론·결정·검증·리스크 중심 `report.md` 를 만든다. 사용자 동의가 있는 `--compact`/`retro --close`/`tidy --apply` 에서는 workbench 문서와 관련 tmp 산출물을 `.sfs-local/archives/` 로 이동해 보존하고, sprint/tmp 표면에는 남겨야 할 durable 문서만 둔다. 히스토리 서사는 `retro.md`, session log, events 가 담당한다. 근거: 2026-05-01 사용자 결정 ("완료 후 실제 작업문서는 보고서로 갱신", "남겨야될것만 남긴다").
24. **Product 배포 = Homebrew + Scoop 양채널 완료** — 사용자가 "배포" / "release publish" / "배포해" 라고 지시하면 stable tag push 뒤 **Homebrew tap (`homebrew-solon-product`) 과 Scoop bucket (`scoop-solon-product`) 을 모두 같은 `v<VERSION>` 으로 갱신·push·검증**해야 완료다. 한 채널만 끝났으면 partial release 로 보고하고 계속 진행한다. 완료 보고 전 최소 확인: product tag 원격 존재, Homebrew formula URL+sha, Scoop manifest URL+hash, 두 channel repo `origin/main` 반영.
25. **동시 세션 Cherry-pick Publish 규율** — `/sfs loop` 나 다른 세션이 같은 branch/worktree 를 갱신 중이면 현재 세션은 **자기가 수행한 변경만** path/hunk 단위로 `git add` 하고, `git diff --cached --name-only` 로 범위를 검증한 뒤 별도 commit/push 한다. unrelated dirty/untracked 는 절대 섞지 않는다. 격리 불가 시 stop + 사용자 결정 대기. 병렬 작업의 마지막 integration 세션이 전체 diff 를 최종 확인하고 clean 상태(또는 명시적 잔여 작업 목록)로 정리한다.
