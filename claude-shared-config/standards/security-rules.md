# Security Rules

## 목적
서비스 전반에 걸쳐 인증, 인가, 입력 검증, 개인정보 보호, 관리자 페이지 보호 기준을 정의한다.

## 1. 인증/인가
- 모든 API는 인증 필요 여부를 명시한다.
- admin API는 일반 사용자 API와 구분된 권한 체계를 사용한다.
- 역할 기반 권한 체크를 코드상 명확히 드러낸다.
- 인증 실패와 인가 실패를 구분해 응답한다.

## 2. 입력 검증
- 모든 외부 입력은 validation을 적용한다.
- path/query/body 값은 형식, 길이, 범위를 검증한다.
- 파일 업로드는 확장자, MIME, 크기 제한을 둔다.
- 신뢰할 수 없는 입력을 로그에 그대로 남기지 않는다.

## 3. 개인정보/민감정보
- 개인정보 원문을 애플리케이션 로그에 남기지 않는다.
- 필요 시 마스킹 규칙을 적용한다.
- 관리자 도구에서도 최소 권한 원칙을 적용한다.
- PII 접근은 명시적 유즈케이스만 허용한다.

## 4. 세션/토큰
- 토큰 저장, 만료, 재발급 정책을 문서화한다.
- 인증 정보는 코드/설정 파일에 하드코딩하지 않는다.
- 비밀정보는 Secret Manager, Parameter Store 등 안전한 저장소를 사용한다.

## 5. 네트워크 및 외부 연동
- 외부 API 호출은 allowlist, timeout, retry 정책을 정의한다.
- callback/webhook은 서명 검증 또는 원천 검증을 수행한다.
- admin endpoint에는 IP 제한 또는 추가 보호수단을 고려한다.

## 6. 보안 로깅
- 인증 실패, 권한 거부, 민감 작업 수행은 감사 로그를 남긴다.
- 로그에는 traceId/correlationId를 포함한다.
- 사용자 비밀번호, 토큰, 주민등록번호 등은 절대 기록하지 않는다.

## 7. 필수 리뷰 포인트
- Broken Access Control
- 입력 검증 누락
- 업로드 취약점
- 민감정보 노출
- 관리자 기능 보호 부재
- rate limit 필요 여부

## 8. 금지 사항
- 편의를 위한 무권한 관리자 API
- 예외 메시지에 내부 시스템 구조 노출
- 민감정보 평문 저장
- 개발 편의용 백도어 유지
